## 三种典型流程

### 流程 A：切换 Provider（不开代理）

```
用户点击 Provider 卡片
  → 前端 invoke("switch_provider", {appId, providerId})
  → commands::provider::switch_provider (commands/provider.rs)
  → services::ConfigService::switch_provider
  → 读 SSOT 中的 Provider.settings_config
  → 调用 atomic_write 覆写 live 配置
      (如 ~/.claude/settings.json)
  → 更新 SSOT 的 current = providerId
  → 发 "provider-switched" 事件
  → 前端 TanStack Query 失效缓存
  → UI 重新渲染，托盘菜单更新
```

### 流程 B：切换 Provider（开代理）

```
用户点击 Provider 卡片（代理模式）
  → 前端 invoke("switch_provider_proxy", {...})
  → commands::proxy::switch_provider_proxy
  → ProxyState::current_providers 更新（内存 + DB）
  → 旧请求正在进行？用 switch_lock 保证切换的原子性
  → 不需要改 live 配置（CLI 指向 127.0.0.1:PORT，不变）
  → 下一个请求到达时，ProviderRouter::select_provider 选出新 Provider
```

**关键差异**：代理模式下切换是**纯内存操作**，速度快、无文件写入；非代理模式要改 live 文件，CLI 需要重启才生效（某些 CLI 会热读）。

### 流程 C：代理模式下的请求 + Failover

```
Claude CLI 发送请求到 127.0.0.1:PORT/v1/messages
  → ProxyServer 手写 hyper accept loop 接收
  → 保留原始 header casing 到 HeaderCaseMap
  → handlers::route_request (proxy/handlers.rs)
  → ProviderRouter::select_provider
     ├─ 检查熔断器状态 (Closed/Open/HalfOpen)
     ├─ Open → 跳过此 Provider，选下一个
     └─ Closed/HalfOpen → 选中
  → providers::claude::forward_request
     ├─ SSE 流式转发
     ├─ 写入 Usage 统计
     └─ 若响应错误 → CircuitBreaker::record_failure
  → 连续失败达阈值 → 打开熔断器 + 触发 FailoverSwitchManager::try_switch
  → 自动切到下一个 Provider
  → 前端收到 "provider-switched" 事件，UI 同步
```

## 五个"第一次接触就困惑"的设计

### 1. 为什么有 `~/.cc-switch/config.json` **又**有 `cc-switch.db`？

SQLite 用于**热数据、查询、索引**；JSON 用于**备份、用户可读、简单数据**。设置项、Proxy 配置、用量聚合都在 SQLite；Provider 的主数据持久到 config.json（并在 DB 有缓存镜像）。见第 23 章。

### 2. 为什么 OpenClaw 不支持 MCP？

OpenClaw 的设计理念不同（类似 Agent SDK 而非 CLI），它有自己的工具系统。cc-switch 按这个事实裁剪 API：`McpApps.openclaw` 字段压根不存在（见第 1 章源码片段）。

### 3. 为什么 Codex 的 Provider 文件命名是 `auth-{name}.json`？

Codex 的 config.toml 里通过 `profile` 引用 auth 配置，profile 名对应独立的 auth 文件。cc-switch 为每个 Provider 生成带 sanitize 名字的 auth 文件。

### 4. `Provider.inFailoverQueue` 为什么是布尔而不是列表？

因为这是 Provider 的属性，不是队列的属性。`FailoverQueue` 是**所有被标记了 `inFailoverQueue=true` 的 Provider 按顺序排队**。见第 16 章。

### 5. 代理为什么要 preserve header case？

Claude CLI 对返回响应的 header casing 有检查（某些库做大小写敏感对比）。如果代理改了 casing，CLI 可能会失败。见第 12 章。
